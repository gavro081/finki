2.2
Why is require(...) placed in the value object constructor rather than in the command handler?
- placing require(...) in the constructor enforces invariants at the very moment the object is created. that makes invalid domain data impossible to construct anywhere in the system, instead of relying on each command handler to remember to validate it.

What is the difference between a value object and an entity in DDD? Give an example from your aggregate of each.
- an entity is defined by a unique identity that persists over time and it can be mutable, whereas a value object is defined strictly by its attributes, not by an identity and it is immutable. examples of an entity and a value object are ThesisProgress and ThesisComment, respectfully.

3.3 QUESTIONS
After sending three update commands and one status command to a single aggregate instance, how many rows exist in DOMAIN_EVENT_ENTRY for that aggregate ID? What does this tell you about how Axon reconstructs the current state?
- there will be 5 rows in DOMAIN_EVENT_ENTRY (1 creation event + 3 update events + 1 status event). this shows that Axon persists every state change as a unique event in the event store. to reconstruct the current state for command processing, axon loads the current snapshot directly from the thesis_progress table using JPA, rather than replaying the 5 events from DOMAIN_EVENT_ENTRY, because I configured GenericJpaRepository. 

Why must the guard check be placed in the @CommandHandler and not in the on(event) method?
- commands represent intent and events represent facts. a command is a request to do something, which can be rejected. therefore, validation/guards belong here to decide if the event should happen. the on(event) method is called when reloading the aggregate, if validation logic is put here that throws an exception, it might break the ability to load a valid existing aggregate from the database/event store. 


4.3 QUESTIONS
Why does @Immutable improve performance here? What does Hibernate skip when it knows an entity will never be modified?
- @Immutable improves performance because Hibernate knows the entity will never be updated, so it can skip dirty checking, snapshot storage for change detection, and flush-time update processing for those instances.

The view and the aggregate map to the same table. What is one advantage and one risk of this design?
- an advantage of mapping the view and aggregate to the same table is simplicity - the read model is easy to implement and always reflects the latest persisted aggregate state without extra projection tables. 
a risk is tighter coupling - the read model becomes tightly tied to the write-side schema, so schema changes or write-side optimizations can easily break query needs or make the read model less efficient/flexible.

TASK 5.1 - Identify the Trigger
when the ThesisProposal aggregate publishes ThesisApprovedEvent, the ThesisProgress context must react by creating a new ThesisProgress aggregate so the student’s thesis work can start being tracked from that point onward. this logic cannot live inside ThesisProposal because that aggregate should only manage the proposal and its approval lifecycle, not the internals of progress tracking. it also cannot live inside ThesisProgress alone, because the trigger for starting progress comes from an event in another aggregate, so the coordination must happen in an external event handler between the two aggregates.

5.3 QUESTIONS
1. `sendAndWait` blocks the event handler thread. What would happen if your handler processed 200 records sequentially with sendAndWait? What alternatives exist?
- If the handler processed 200 records sequentially with sendAndWait, it would block the event-handler thread until all 200 commands finished, which increases latency and can reduce throughput for the whole processor. slternatives include splitting work into smaller batches, using send(...) asynchronously, publishing a follow-up process event for parallel workers, or modeling the coordination by using the saga pattern as mentioned in the third auditory exercise.

2. Why is it wrong to inject and call the aggregate's Axon repository directly inside an event handler instead of going through the command gateway?
- the command gateway enforces routing, validation, interceptors, auditing, and aggregate consistency rules. if the handler loaded aggregates directly and called methods itself, it would bypass the normal command-processing lifecycle that CQRS and Axon are built around.
